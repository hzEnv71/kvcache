# KVCache 项目面经整理

> 目标：围绕你当前这个 KVCache 项目的真实实现，整理一套“从项目介绍 -> 核心设计 -> 底层机制 -> 故障排查 -> 扩展优化”的面试问答。内容尽量按面试官会追问的方式展开，帮助你在面试里做到“先讲清，再讲深，再讲取舍”。

---

# 1. 项目一句话介绍

可以先用这版：

> 这是一个基于 Go 实现的分布式缓存系统，节点之间通过 gRPC 通信，使用 etcd 做服务注册发现，使用一致性哈希做 key 到 owner 节点的路由，缓存层支持 LRU / LRU-2、TTL 过期、singleflight 防击穿，并且我把读写链路改成了严格 owner-only 模式，避免多节点副本残留和删除不一致问题。

如果面试官让你再展开一点，可以说：

> 整体上它分成控制面和数据面。控制面用 etcd 维护节点注册、下线感知和成员变更；数据面由 Group、Cache、Store、PeerPicker、gRPC client/server 组成。请求进来以后，先根据一致性哈希选 owner，非 owner 只负责转发，owner 负责真正的读写和回源。这样能把分布式路由、节点发现、缓存淘汰和并发控制串起来。

# 2. 面试官高频开场问题

## 2.1 你这个项目解决的核心问题是什么？

可以答：

> 核心是把单机缓存扩展成分布式缓存。单机缓存只能解决本机热点数据命中问题，而分布式场景下会有多个服务节点，必须解决几个问题：第一，key 该落到哪台机器；第二，节点上下线后路由如何变化；第三，并发 miss 时如何避免击穿；第四，缓存如何做过期和淘汰；第五，读写语义怎么定义。我这个项目就是围绕这些问题搭了一套比较完整的实现。

## 2.2 你这个项目的核心亮点是什么？

可以答：

> 我觉得有几个亮点。第一，用 etcd + watcher 做服务发现，而不是把节点列表写死；第二，用一致性哈希做 owner 路由，避免节点变化时大规模迁移；第三，用 strict owner-only 统一了 Get/Set/Delete 语义，避免之前本地副本残留的问题；第四，用 singleflight 避免热点 key 并发 miss 导致的缓存击穿；第五，底层 store 既支持 LRU，也支持两级 LRU-2，并且支持 TTL 和后台清理。

# 3. 你要能讲清楚的系统架构

## 3.1 整体架构怎么分层？

推荐这样答：

> 我把它分成 4 层。第一层是接入层，也就是 cmd/client 和 server.go，对外提供 gRPC 接口；第二层是缓存协调层，也就是 group.go，负责 Get/Set/Delete 主流程、owner 路由、singleflight 和回源；第三层是节点管理层，也就是 peers.go + consistenthash，负责服务发现、哈希环和远端 client 管理；第四层是本地存储层，也就是 cache.go + store 目录，负责本地缓存读写、淘汰和 TTL 过期。

## 3.2 为什么要引入 etcd？

推荐答法：

> etcd 在这个项目里不是存业务缓存，而是做控制面。它负责维护节点是否在线、节点地址是什么、什么时候新增、什么时候下线。缓存数据本身还是存在各个缓存节点内存里。也就是说：KVCache 是数据面，etcd 是控制面。

如果面试官追问“为什么不用 etcd 直接存缓存值”，可以说：

> etcd 适合存少量强一致元数据，不适合高频热点业务读写。它底层是 Raft，写成本高，吞吐和延迟目标跟缓存系统不同。

# 4. 请求主链路：一定会被问

## 4.1 Get 链路怎么走？

推荐答：

> Get 先进入 server.go 的 gRPC Handler，然后根据 group name 拿到对应的 Group。接着 Group.Get 会先判断是否启用了 peers。如果没有 peers，就退化成单机缓存模式：先查本地缓存，miss 后走 load。若启用了 peers，就先通过 PickPeer(key) 用一致性哈希找到 owner。owner 如果是远端，就直接调用 peer.Get 去 owner 节点读；如果 owner 是本机，就查本地缓存。只有 owner 本机 miss 的时候，才会调用 load -> singleflight -> loadData -> getter 回源。也就是说，当前版本下 owner 节点 miss 后只回源，不会再继续找其他 peer。

## 4.2 Set 链路怎么走？

推荐答：

> Set 也是先进入 Group.Set。先做参数校验，再判断请求是不是 peer 转发来的。如果 metadata 中有 from_peer=true，说明这个请求已经被转发到 owner 了，就只做本地写，不再继续路由。如果没有 peers，就按单机模式直接本地写。如果有 peers，就先 PickPeer(key) 找 owner：owner 是远端就转发给远端 owner；owner 是本机就直接本地写。这里我把写链路改成了 strict owner-only，所以非 owner 不会保留业务副本。

## 4.3 Delete 链路怎么走？

推荐答：

> Delete 和 Set 对齐。也是先判断是不是 peer 请求；如果是，就本地删；如果没有 peers，就单机本地删；如果启用了 peers，就 PickPeer(key) 找 owner。owner 是远端就转发到远端删，owner 是本机就本地删。这样删除链路也变成了 owner-only，不需要清理多个残留副本。

# 5. 面试官很爱追问：为什么要做 strict owner-only

## 5.1 你为什么从“本地先写再同步”改成 owner-only？

推荐答：

> 因为本地先写再同步 owner 其实是最终一致模型，会导致一个 key 在多个节点上短暂存在副本。这样读写看似快，但会引入副本残留、删除不完整、故障后语义混乱的问题。后来我把它改成 strict owner-only：Get/Set/Delete 都先找 owner，非 owner 只转发，owner 才真正执行。这样虽然多了一次 hop，但语义清晰很多，也更适合面试里解释一致性和路由。

## 5.2 owner-only 的缺点是什么？

答：

> 主要有几个：第一，非 owner 节点的请求会多一次网络 hop；第二，owner 节点压力更集中；第三，如果没有副本，owner 挂了之后这个 key 只能依赖重新路由和回源恢复；第四，对成员视图一致性要求更高，因为所有节点必须基于同一个 ring 算 owner。

## 5.3 那为什么你最后还是选 owner-only？

答：

> 因为这个项目是缓存系统，不是强一致分布式数据库。相比做复杂副本同步，我更关注路由和语义清晰。owner-only 很适合作为当前阶段的设计：它能把服务发现、一致性哈希、peer 路由、TTL、singleflight 这些核心点讲完整，而且代码更干净。

# 6. 一致性哈希：必问底层

## 6.1 为什么要用一致性哈希？

答：

> 因为普通取模在节点数变化时会导致大规模 key 重映射，而一致性哈希只会影响局部区间，扩缩容代价更小。

## 6.2 你的哈希环底层怎么实现？

答：

> 我维护了一个有序的虚拟节点哈希数组 `keys`，以及一个 `hashMap[virtualHash] -> realNode` 映射。每个真实节点会展开成多个虚拟节点，插入到环上。Get(key) 时先对 key 哈希，然后在有序 `keys` 上用二分查找找第一个大于等于 keyHash 的虚拟节点，如果越界就回绕到 0，最后再映射回真实节点。

## 6.3 为什么需要虚拟节点？

答：

> 虚拟节点能让数据分布更均匀，否则真实节点数少的时候容易倾斜。另外后续如果想做权重控制，也可以通过调整每个真实节点的虚拟节点数实现。

## 6.4 你这个实现里 `keys` 为什么必须有序？

答：

> 因为 Get(key) 用的是二分查找，二分查找要求底层数组有序。所以 Add 节点之后一定要 `sort.Ints(m.keys)`。

## 6.5 你遇到过什么一致性哈希相关的 bug？

强烈建议用你的真实经历答：

> 我一开始把 self 节点排除在 hash ring 外，只把远端 peer 放进 ring。结果每个节点都基于不同的成员集算 owner，同一个 key 在不同节点上得到不同 owner。后来我改成：所有节点都进 `consHash`，但只有远端节点才进 `clients`。这样 ring 视图才能全局一致。

# 7. PeerPicker / ClientPicker：很容易被深挖

## 7.1 PeerPicker 是干什么的？

答：

> PeerPicker 是 owner 路由抽象。给它一个 key，它返回三个信息：peer、ok、self。peer 表示远端 client；ok 表示有没有找到可用 owner；self 表示 owner 是否为本机。Group 的 Get/Set/Delete 都靠它决定是本地执行还是远端转发。

## 7.2 为什么 PickPeer 要返回 `(peer, ok, self)`，而不是只返回 peer？

答：

> 因为 owner 可能是本机。对本机 owner 来说，不需要也不应该通过 gRPC 调自己，所以要显式区分 self。之前我就踩过一个坑：owner 是自己时还去 `clients` 里找连接，但 `clients` 本来不存自己，结果报 `owner peer unavailable`。后来我改成 owner=self 时直接 `ok=true, self=true`。

## 7.3 `clients` 和 `consHash` 为什么要分开？

答：

> `consHash` 表示的是路由视图，也就是哪些节点参与 owner 计算；`clients` 表示的是远端连接视图，也就是当前节点有哪些 gRPC client 可以直接调用。self 节点应该在 `consHash` 里，但不应该在 `clients` 里。

# 8. etcd 服务发现：高频深挖点

## 8.1 你的节点注册是怎么做的？

答：

> 每个节点启动后会调用 Register，把 `/services/{svc}/{addr}` 写入 etcd，并绑定一个 lease。之后通过 KeepAlive 保持 lease 在线。正常退出时会主动 revoke lease；异常宕机时则依赖 lease TTL 到期后被 etcd 自动删除。

## 8.2 为什么新增节点能很快感知到，下线节点却慢？

答：

> 因为新增是一个 Put 事件，注册完成后 etcd 会立刻把 Put 推给 watcher；而异常下线不是主动 Delete，而是 lease 不再续约，要等 TTL 到期 etcd 才会删除这个 key。所以节点新增感知快，节点宕机感知慢。

## 8.3 你这块遇到过什么 bug？

可以答你真实踩坑：

> 我一开始发现节点宕机后，过了 lease TTL 仍然会继续路由到下线节点。最后定位到是 watch Delete 事件处理错了：我一直读 `event.Kv.Value`，但 Delete 事件里这个值通常为空，真正被删除的旧值要从 `event.PrevKv.Value` 里拿。而且 watch 还要加 `WithPrevKV()`，不然 Delete 事件连旧值都拿不到。修完之后，节点下线后才能正确从 hash ring 里移除。

## 8.4 如果面试官问：为什么不用主动健康检查替代 lease？

答：

> lease 的优点是实现简单，和 etcd 控制面天然结合；缺点是下线感知速度受 TTL 影响。如果想进一步优化，可以在 lease 基础上叠加主动健康检查或者应用层探测，做到更快摘除和更少误判。

---

# 9. gRPC / metadata：很容易被问到“你为什么这样设计”

## 9.1 为什么要用 gRPC metadata 传 `from_peer`？

答：

> 因为 `context.WithValue` 只在当前进程内有效，不能自动跨 gRPC 传到远端。我要区分“这是客户端直接请求”还是“这是 peer 转发请求”，所以必须把 `from_peer=true` 放在 outgoing metadata 里，远端再通过 incoming metadata 读取。

## 9.2 如果不用 metadata 会发生什么？

答：

> 远端收不到 `from_peer` 标记，就会把一个 peer 转发请求误判成普通客户端请求，然后再次进入 owner 路由，轻则多一次跳转，重则形成递归转发、超时或者错误的 owner unavailable。

## 9.3 为什么 peer 请求到达 owner 后只做本地处理？

答：

> 因为那说明路由已经完成了，owner 节点就是最终处理者。如果它再继续路由，就违背了 owner-only 设计，还会导致循环调用或者额外 hop。

---

# 10. singleflight：面试官很喜欢问

## 10.1 你为什么要引入 singleflight？

答：

> 为了防止缓存击穿。如果某个热点 key 在某一刻本地 miss，没有 singleflight 的话会有很多 goroutine 同时去回源 getter，造成瞬时放大。用了 singleflight 以后，同一个 key 的并发 miss 只会触发一次实际加载，其他请求复用结果。

## 10.2 singleflight 用在你项目的哪一层？

答：

> 用在 `Group.load()` 里。只有 owner 本机 miss 时，才会进入 `singleflight.Group.Do(key, fn)`。这保证了真正的回源只在 owner 节点发生，而且同一时间只做一次。

## 10.3 它解决不了什么问题？

答：

> 它只能合并“同一节点、同一 key、同一时间窗口”的并发请求，不能解决跨节点的重复回源，也不能替代缓存淘汰、分布式锁或一致性协议。

---

# 11. TTL、过期与清理

## 11.1 你这个项目支持 TTL 吗？

答：

> 支持。当前实现主要是 Group 级默认 TTL，通过 `WithExpiration(...)` 配给整个 Group。写入缓存时会根据 `g.expiration` 走 `AddWithExpiration`，底层 store 会保存过期时间。

## 11.2 过期是怎么生效的？

答：

> 两层机制：第一层是惰性过期，也就是 Get 时发现 key 已经过期，就视为 miss 并删除；第二层是后台定期清理，LRU 和 LRU-2 存储实现里都有 cleanup loop，会周期性把过期项扫掉。

## 11.3 如果面试官问：为什么同一个组在不同节点上不能设置不同 TTL？

答：

> 因为当前 TTL 是 Group 级配置，不是 key 级元数据。如果同名 Group 在不同节点上 expiration 不一致，那么最终哪个 TTL 生效取决于 owner 在哪台机器，会导致语义不一致。所以在分布式模式下，同名 Group 的配置必须保持一致。如果要支持不同 key 的不同 TTL，应该把 TTL 作为请求参数和数据属性一起传给 owner。

---

# 12. LRU / LRU-2：存储层追问

## 12.1 你为什么做了两种存储实现？

答：

> LRU 简单、通用，但容易被偶发扫描流量污染。LRU-2 的思路是第一次访问先进入候选区，第二次命中后再提升到热点区，这样能更好地区分偶发访问和稳定热点。所以我保留了两种实现，便于比较策略差异。

## 12.2 LRU-2 的底层怎么做的？

答：

> 我这里实现的是两级缓存模型。一级缓存存第一次访问的数据，二级缓存存再次命中的热点数据。并且做了分桶，每个 bucket 一把锁，减少全局锁竞争。TTL 也是按节点上的 `expireAt` 字段判断，并配合 cleanup loop 定期清理。

# 13. 命中率和统计：你现在可以怎么答

## 13.1 你怎么观测缓存效果？

答：

> 我在 Group 里维护了一套统计项，包括 `local_hits/local_misses/peer_hits/peer_misses/loads/loader_hits/loader_errors/loadDuration` 等，并提供了 `Stats()` 方法。这样可以看到本地命中、远端命中、回源次数和平均回源耗时。我还在退出时把 stats 写到日志文件，方便离线分析。

## 13.2 命中率怎么算？

答：

> 当前实现里有一个基础命中率，主要按 `local_hits / (local_hits + local_misses)` 计算。另外也能通过 `peer_hits/peer_misses` 看跨节点命中情况。严格来说，如果要做全局集群命中率，还需要聚合多节点统计。

# 14. 故障和真实排障：这部分最加分

这块建议你一定要讲自己真实踩过的坑，因为这会让面试官觉得你不是“照着教程写”，而是真正做过系统调试。

## 14.1 你排过哪些典型 bug？

你可以按这个顺序讲：

### Bug 1：不同节点算出的 owner 不一致
- 原因：把 self 排除在 hash ring 外
- 现象：同一个 key 在不同节点算出来的 owner 不同
- 修复：所有节点都放进 `consHash`，只有远端节点放进 `clients`

### Bug 2：peer 标记一直失效
- 原因：一开始用了 `context.WithValue("from_peer")`
- 现象：远端节点拿不到标记，误判为普通客户端请求
- 修复：改成 gRPC metadata

### Bug 3：owner 是自己却报 `owner peer unavailable`
- 原因：PickPeer 在 self 场景下还去 `clients` 里找连接
- 现象：明明 owner 是本机，却因为 `clients` 不存自己而报错
- 修复：self 场景直接返回 `ok=true, self=true`

### Bug 4：节点宕机后仍然路由到死节点
- 原因：Delete 事件读错字段，只读了 `event.Kv.Value`
- 现象：lease 过期后节点仍残留在 ring
- 修复：watch 加 `WithPrevKV()`，Delete 时读 `event.PrevKv.Value`

### Bug 5：owner 节点 miss 后又继续 peer 路由
- 原因：`loadData()` 里保留了“先找 peer”逻辑
- 现象：请求已经到达 owner，本地 miss 后还继续转发，导致 owner unavailable
- 修复：strict owner-only 下 `loadData()` 只走 getter 回源，不再继续找 peer

# 15. 面试官可能进一步刨根问底的问题

## 15.1 如果 owner 挂了怎么办？

答：

> 当前项目是缓存系统，不做强副本恢复。owner 挂了之后，控制面通过 etcd lease 过期感知节点下线，剩余节点会把它从 ring 里移除，之后请求会路由到新的 owner。因为这是缓存而不是数据库，所以数据丢了也没关系，新 owner miss 后会重新回源 getter 填充。

## 15.2 如果面试官说：那这不是会丢缓存吗？

答：

> 是的，这个系统目前接受“节点故障导致缓存丢失”的语义。因为缓存本来就是可丢的，重点是保证路由重新收敛后还能继续服务。如果要做更强的高可用，可以在 owner-only 基础上增加副本或者持久化。

## 15.3 如果要继续演进，你会怎么做？

答：

> 我会分三步。第一步，对服务发现和路由做更稳的状态同步，比如统一 ring 版本；第二步，增加 key 级 TTL 和更清晰的配置/观测接口；第三步，如果目标是更高可用，可以在 owner-only 基础上加副本节点，做主备或多副本。

# 16. 你可以直接背的“项目亮点总结”

可以直接这样说：

> 这个项目我主要做了三类工作。第一类是架构梳理，把整个缓存系统拆成 gRPC 接入层、Group 缓存协调层、PeerPicker 路由层和 Store 本地存储层；第二类是分布式一致性语义收敛，把原来本地先写再同步的模式改成 strict owner-only，让 Get/Set/Delete 都先找 owner，非 owner 只转发，owner 真正执行；第三类是问题排查和稳定性修复，比如修复了一致性哈希 ring 不一致、peer metadata 丢失、Delete 事件读错字段、owner miss 后继续 peer 路由等问题。

2. 你们在服务发现上是用 etcd / consul 还是自研控制面？
3. 你们在缓存命中率、TTL、热点 key 和故障摘除上最关注哪些指标？
4. 如果是强一致场景，你们会怎么在缓存层和数据库之间做权衡？

这样会显得你不是只会写代码，而是能站在系统设计和生产场景角度思考。

---

# 18. 最后给你的面试建议

真正面试时，不要一上来就陷进代码细节。建议按这个节奏：

1. **先一句话讲项目做什么**
2. **再讲整体架构分层**
3. **再挑一条主链路讲清楚（Get/Set/Delete）**
4. **最后讲你修过的坑和 trade-off**

这样面试官会觉得你：
- 先有全局观
- 再能下钻到实现
- 最后还能讲问题和取舍

这就是最加分的表达方式。
